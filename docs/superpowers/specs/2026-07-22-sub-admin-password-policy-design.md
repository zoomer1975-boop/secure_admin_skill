# 부관리자 비밀번호 정책 설계

## 목적

초대받은 부관리자가 최종 비밀번호를 설정할 때 강한 비밀번호 정책을 서버에서 강제한다. 동일한 검증을 이후의 비밀번호 변경과 재설정에도 적용해 우회 경로를 만들지 않는다.

## 선택한 접근

기존 `skills/building-secure-admin-pages/SKILL.md`의 `Accounts and authentication` 계약에 비밀번호 정책을 추가한다. 별도 정책 계층이나 새 도구는 만들지 않는다. 애플리케이션 구현 시 기존 서버 측 비밀번호 검증 지점 하나를 재사용하거나 확장하도록 지시한다.

## 비밀번호 정책

- 길이는 10자 이상이어야 한다.
- ASCII 대문자 `[A-Z]`, ASCII 소문자 `[a-z]`, 숫자 `[0-9]`, 특수문자를 각각 하나 이상 포함해야 한다.
- 특수문자는 공백을 제외한 ASCII 출력 기호로 정의한다.
- 비교 전에 비밀번호, 로그인 ID, 이름에 Unicode NFKC 정규화, 대소문자 구분 제거, 공백 제거를 적용한다. 정규화된 비밀번호에 정규화된 로그인 ID 또는 이름이 포함되면 거부한다.
- "일부"는 연속된 3자 이상의 ID 또는 이름 조각으로 정의한다. ID 또는 이름이 1~2자라면 전체 값이 포함된 경우 거부한다.
- 같은 문자가 3회 이상 연속되면 거부한다. 예: `aaa`, `111`.
- 오름차순 또는 내림차순 숫자가 3자리 이상 연속되면 거부한다. 예: `123`, `321`.

## 적용 흐름과 실패 처리

- 초대 완료, 비밀번호 변경, 비밀번호 재설정에서 같은 정책을 적용한다.
- 클라이언트 검사는 안내용으로만 사용할 수 있으며, 서버가 해시 생성과 저장 전에 최종 검사를 수행한다.
- 실패 응답은 충족하지 못한 규칙을 알려 주되 입력된 비밀번호를 반환하거나 기록하지 않는다.
- 정책 위반 시 계정 상태, 초대 상태, 비밀번호 해시를 변경하지 않는다.
- 기존 초대 방식은 유지한다. 등록자는 최종 비밀번호를 알거나 대신 지정할 수 없다.

## 변경 범위

- `skills/building-secure-admin-pages/SKILL.md`: 계정 및 인증 계약에 정책과 서버 측 적용 범위를 추가한다.
- `test/install.test.mjs`: 설치된 `SKILL.md`에 핵심 정책 문구가 포함되는지 기존 테스트에 최소 assertion을 추가한다.
- README, 설치기, 패키지 구조는 변경하지 않는다.

## 검증

- 설치된 스킬이 10자 이상, 네 종류 문자 조합, ID/이름 포함 금지, 3회 반복문자 금지, 3자리 연속숫자 금지를 모두 명시하는지 확인한다.
- `npm test`로 Codex 및 Claude Code 설치 흐름과 새 정책 문구 배포를 검증한다.
- `npm pack --dry-run`으로 배포 패키지에 수정된 스킬이 포함되는지 확인한다.

## 유지되는 전체 보안 계약

이번 변경은 아래 기존 계약을 축소하거나 대체하지 않는다.

### Address

- 같은 도메인의 공개 링크가 없는 무작위 경로를 사용한다.
- `admin`, `administrator`, `manager`, `master`, `backoffice`, `cms` 같은 의미 기반 경로를 사용하지 않는다.
- 공개 내비게이션, 사이트맵, robots 항목, 클라이언트 로그에 경로를 노출하지 않는다.
- 존재하지 않는 관리자 경로는 일반 404를 반환한다.

### Accounts and authentication

- 한 사람마다 하나의 기명 계정을 발급하며 계정을 공유하지 않는다.
- 여러 명의 기명 최고관리자와 부관리자를 지원한다.
- 마지막 활성 최고관리자를 삭제, 비활성화, 강등하지 못하게 한다.
- 최고관리자 변경에는 비밀번호와 TOTP 재인증을 요구한다.
- 30분 동안 유효한 일회용 초대를 생성해 등록자에게 한 번만 표시하고 별도 보안 채널로 전달하게 한다.
- 초대받은 관리자가 최종 비밀번호와 TOTP 비밀값을 직접 설정하며 등록자는 최종 비밀번호를 알 수 없다.
- 비밀번호는 복원 불가능한 해시로 저장하고 인증정보는 HTTPS로만 전송한다.
- 모든 관리자에게 TOTP를 요구하고 반복 인증 실패를 제한한다.
- 비활성화, 비밀번호 변경, TOTP 재설정 후 모든 세션을 폐기한다.
- 비밀번호와 TOTP로 최근 재인증한 최고관리자만 다른 관리자의 TOTP를 재설정할 수 있으며 대상 계정의 모든 세션을 폐기한다.
- 본 설계의 비밀번호 정책을 초대 완료, 변경, 재설정 시 서버에서 강제한다.

### Access IPs

- 모든 계정에 IP 허용 목록을 요구하고 여러 IPv4, IPv6, CIDR 항목을 지원한다.
- 허용되지 않은 주소는 로그인 화면 표시 전에 거부하고 시도를 감사 기록한다.
- 명시적으로 신뢰한 프록시의 전달 주소 헤더만 신뢰한다.

### Authorization

- 모든 메뉴와 하위 메뉴에 `read`, `create`, `update`, `delete`, `export` 권한을 각각 정의한다.
- 모든 서버 요청에서 권한을 검사하며 메뉴 숨김을 권한 검사로 간주하지 않는다.
- 부관리자는 자신이 보유하지 않은 권한을 다른 계정에 부여할 수 없다.
- 위임받은 부관리자 관리자는 자신의 권한 범위 안에서만 계정을 조회, 생성, 수정할 수 있다.
- 부관리자의 비활성화와 삭제는 최고관리자만 수행할 수 있고 이 권한은 위임할 수 없다.

### Administrator management

- ID, 이름, 선택 메모, 메뉴별 행위 권한, IP 허용 목록만 수집한다.
- 등록 화면에 비밀번호 또는 이메일 입력란을 추가하지 않는다.
- 메모에 민감정보나 불필요한 개인정보를 적지 않도록 경고한다.
- 변경되지 않는 내부 계정 식별자를 사용하고 삭제된 로그인 ID를 다른 사람에게 재사용하지 않는다.

### Audit log

- 인증, 로그아웃, 실패, IP 거부, `CREATE`, `READ`, `UPDATE`, `DELETE`, 권한 변경, 계정 상태 변경, 다운로드, 내보내기를 포함한 모든 관리자 행위를 기록한다.
- 행위자 식별자, 시각, 출발 IP, 대상 유형과 식별자, 행위 분류, 이벤트 이름, 성공 여부를 기록한다.
- 비밀번호, TOTP 비밀값, 세션 또는 초대 토큰, 전체 개인정보 payload를 기록하지 않는다.
- 로그는 분리되고 변조에 강하게 보관하며 어떤 관리자도 수정하거나 삭제할 수 없다.
- 최고관리자는 조회와 내보내기를 할 수 있고 명시적으로 허가된 부관리자는 조회만 할 수 있다.
- 감사 기록 작성에 실패하면 관리자 행위를 거부한다.

### Privacy and retention

- 관리자 이름, IP, 메모, 접속기록의 목적, 법적 근거, 항목, 보유기간, 파기 방법을 문서화한다.
- 필요한 정보만 수집하고 필요가 끝나면 복원할 수 없게 파기한다.
- 계정 비활성화와 세션 폐기는 즉시 수행하고 감사 또는 법률상 필요한 기록만 유지한다.
- 권한 부여, 변경, 회수 이력을 최소 3년 보관한다.
- 접속기록은 최소 1년 보관하고, 5만 명 이상의 정보주체 또는 민감정보·고유식별정보를 처리하는 등 현행 법령상 요구되면 최소 2년 보관한다.
- 내부 관리계획에 접속기록과 다운로드 검토 주기, 방법, 후속 조치를 정한다.
- 개인정보 처리방침, 개인정보 보호 책임자 또는 담당부서, 침해 대응과 통지 절차를 확인한다.

### Failure behavior

- 허용되지 않은 IP에는 로그인 화면을 보여 주지 않는다.
- 만료되거나 이미 사용된 초대에는 같은 일반 오류를 사용한다.
- 권한 위반은 서버에서 거부하고 내부 권한 구조를 노출하지 않은 채 감사 기록한다.
- 업무상 전체 값이 필요하지 않으면 관리자 목록과 감사 화면의 개인정보를 마스킹한다.

## 승인 기준

- 위 정책 문구가 기존 초대 계약과 충돌하지 않는다.
- 비밀번호 입력란을 관리자 등록 화면에 추가하지 않는다.
- 서버 측 단일 검증 지점을 요구해 초대, 변경, 재설정 경로가 같은 정책을 사용한다.
- 정책 배포 여부를 기존 설치 테스트의 한 assertion으로 검증한다.
