# 안전한 관리자 페이지 제작 규칙 스킬 설계

## 1. 목적

웹사이트의 관리자 페이지를 새로 만들거나 수정·검토할 때 공통으로 적용할 보안, 권한, 감사로그, 개인정보 보호 규칙을 Codex 스킬로 제공한다.

이 스킬은 특정 프레임워크의 관리자 화면을 생성하는 템플릿이 아니다. 프로젝트의 기존 인증·라우팅·데이터 계층을 우선 재사용하면서 필수 보안 조건을 빠뜨리지 않도록 지시하는 규칙 스킬이다.

## 2. 산출물

- 스킬명: `building-secure-admin-pages`
- 설치 위치: `C:\Users\Owner\.codex\skills\building-secure-admin-pages`
- 기본 구성:
  - `SKILL.md`: 적용 절차, 필수 규칙, 검증 체크리스트
  - `agents/openai.yaml`: 스킬 표시 이름과 기본 호출 문구
- 제외 사항:
  - 프레임워크별 코드 템플릿
  - 별도 라이브러리 또는 실행 스크립트
  - 법률 전문 복사본

현행 법령의 세부 문구는 스킬에 고정하지 않는다. 법적 적합성 판단이 필요한 프로젝트에서는 개인정보보호위원회와 국가법령정보센터의 최신 공식 자료를 다시 확인하도록 지시한다.

## 3. 적용 시점

다음 작업에 스킬이 적용되어야 한다.

- 웹사이트 관리자 페이지 신규 제작
- 관리자 인증·인가·계정 관리 기능 추가 또는 변경
- 관리자 메뉴와 접근권한 설계
- 개인정보를 처리하는 백오피스 또는 운영 도구 검토
- 관리자 감사로그, 접속기록, 다운로드 기록 구현

## 4. 작업 흐름

1. 프로젝트의 기존 인증, 라우팅, 세션, 권한, 감사로그 구현을 확인한다.
2. 사이트에 필요한 관리자 메뉴와 각 메뉴의 하위 메뉴를 확정한다.
3. 관리자 화면에서 처리하는 개인정보 종류와 규모를 확인한다.
4. 이 문서의 불변 규칙을 기존 구조에 최소 변경으로 적용한다.
5. 서버 측 권한검사와 감사로그를 먼저 검증한 뒤 화면 노출을 검증한다.
6. 개인정보 처리방침, 보유기간, 파기 및 사고 대응 항목을 점검한다.

## 5. 불변 규칙

### 5.1 관리자 주소

- 관리자 페이지는 홈페이지에서 링크하지 않는다.
- 같은 도메인의 무작위 경로를 사용한다.
- `admin`, `administrator`, `manager`, `master`, `backoffice`, `cms`처럼 역할을 유추할 수 있는 경로를 사용하지 않는다.
- 관리자 경로를 공개 메뉴, 사이트맵, robots 목록, 클라이언트 로그에 노출하지 않는다.
- 무작위 경로는 보조 수단일 뿐 인증수단으로 간주하지 않는다.

### 5.2 관리자 계정과 역할

- 계정 공유를 금지하고 1인 1계정을 적용한다.
- 계정마다 변경되지 않는 내부 식별자를 부여하고, 삭제된 아이디를 다른 사람에게 재사용하지 않는다.
- 역할은 최고관리자와 부관리자로 구분한다.
- 복수의 최고관리자를 허용한다.
- 마지막 활성 최고관리자는 삭제, 비활성화 또는 강등할 수 없다.
- 최고관리자 추가·삭제·강등에는 현재 비밀번호와 TOTP 재인증을 요구하고 감사로그를 남긴다.
- 부관리자 삭제·비활성화는 최고관리자만 수행할 수 있다.

### 5.3 초대와 인증

- 계정 생성자가 대상 관리자의 최종 비밀번호를 입력하거나 알 수 없게 한다.
- 계정 생성 후 30분 동안 한 번만 사용할 수 있는 초대 링크를 화면에 한 번만 표시한다.
- 초대 링크는 사용 또는 만료 즉시 폐기한다.
- 대상 관리자는 초대 링크에서 직접 비밀번호와 TOTP를 등록한다.
- 비밀번호는 복호화할 수 없는 안전한 단방향 방식으로 저장한다.
- 최고관리자와 부관리자 모두 TOTP를 필수로 사용한다.
- TOTP 재설정은 재인증한 최고관리자만 수행하고 대상 계정의 기존 세션을 모두 폐기한다.
- 로그인 반복 실패 시 일시적으로 접근을 제한한다.
- 인증정보와 개인정보는 HTTPS로만 송수신한다.

### 5.4 접근 IP

- 모든 관리자 계정에 접근 IP 허용 목록을 적용한다.
- IPv4, IPv6, CIDR 대역을 여러 개 등록할 수 있게 한다.
- 허용 목록에 없는 주소의 요청은 인증 전에 거부하고 시도 내역을 기록한다.
- IP 비교와 CIDR 판정은 신뢰할 수 있는 서버 측 주소를 기준으로 수행한다.
- 프록시 헤더는 신뢰할 프록시가 명시된 경우에만 사용한다.

### 5.5 권한

- 사이트별 관리자 메뉴와 모든 하위 메뉴를 권한 목록에 표시한다.
- 각 메뉴마다 `조회`, `추가`, `수정`, `삭제`, `내보내기` 권한을 독립적으로 선택한다.
- 화면의 메뉴 숨김과 별개로 모든 요청에서 서버 측 권한검사를 수행한다.
- 부관리자는 자신이 보유한 범위를 초과하는 권한을 다른 부관리자에게 부여할 수 없다.
- 부관리자 관리 권한이 있는 부관리자는 허용 범위 안에서 계정 조회·추가·수정만 수행할 수 있다.
- 부관리자 삭제·비활성화 권한은 체크박스로 위임할 수 없다.

### 5.6 부관리자 관리 메뉴

필수 입력 및 설정 항목은 다음과 같다.

- 아이디
- 이름
- 메모
- 메뉴별 행위 권한
- 접근 IP 허용 목록

비밀번호 입력란과 이메일 입력란은 두지 않는다. 메모에는 민감정보나 불필요한 개인정보를 입력하지 않도록 안내한다.

### 5.7 감사로그 메뉴

관리자 페이지에서 발생하는 모든 행위를 기록한다.

- 기본 분류: `CREATE`, `READ`, `UPDATE`, `DELETE`
- 추가 분류: 인증, 로그아웃, 인증 실패, 권한 변경, 계정 상태 변경, 내보내기, 다운로드
- 필수 기록값: 행위자 계정 식별자, 일시, 접속 IP, 대상 종류와 식별자, 행위 분류, 구체적 행위, 성공·실패 결과
- 기록 금지값: 비밀번호, TOTP 비밀값, 세션·초대 토큰, 전체 개인정보 원문
- 로그는 일반 관리자 데이터와 분리하고 위·변조와 임의 삭제를 방지한다.
- 최고관리자는 조회와 내보내기를 할 수 있다.
- 감사로그 조회 권한을 받은 부관리자는 조회만 할 수 있다.
- 어떤 관리자도 감사로그를 수정하거나 삭제할 수 없다.
- 감사기록을 남길 수 없는 경우에는 조회를 포함한 관리자 요청을 완료하지 않는다.

## 6. 개인정보 보호 규칙

- 관리자 이름, 접속 IP, 메모, 접속기록의 처리 목적과 법적 근거를 정한다.
- 목적에 필요한 최소 정보만 수집한다.
- 항목별 보유기간과 파기 방법을 정한다.
- 계정 삭제 요청은 즉시 비활성화와 세션 폐기로 처리하고, 보존 의무가 끝난 개인정보는 복구할 수 없게 파기한다.
- 권한 부여·변경·말소 내역은 최소 3년 보관한다.
- 개인정보처리시스템 접속기록은 원칙적으로 1년 이상 보관한다.
- 5만 명 이상의 정보주체 정보를 처리하거나 고유식별정보·민감정보를 처리하는 등 법정 조건에 해당하면 접속기록을 2년 이상 보관한다.
- 접속기록과 개인정보 다운로드 상황의 점검 주기, 방법, 사후조치 절차를 내부 관리계획에 포함한다.
- 개인정보 처리방침, 개인정보 보호책임자 또는 담당부서, 유출 사고 대응·통지 절차를 확인한다.

이 스킬은 개발 규칙을 제공하며 개별 사업에 대한 법률 자문을 대신하지 않는다.

## 7. 요청 처리와 오류 규칙

- 존재하지 않는 관리자 경로는 일반적인 404 응답으로 처리한다.
- 허용되지 않은 IP는 로그인 화면을 제공하지 않고 거부한다.
- 인증되지 않은 요청과 권한이 부족한 요청을 구분하되 내부 권한 구조를 응답에 노출하지 않는다.
- 만료·사용된 초대 링크에는 동일한 일반 오류 메시지를 표시한다.
- 권한 초과 요청은 UI 상태와 무관하게 서버에서 거부하고 감사로그에 남긴다.
- 계정 비활성화, 비밀번호 변경, TOTP 재설정 시 기존 세션을 모두 폐기한다.
- 관리자 목록과 감사로그 화면에서 개인정보를 불필요하게 전체 노출하지 않는다.

## 8. 검증 기준

스킬 자체는 다음 시나리오에서 잘못된 구현을 거부하거나 수정하도록 유도해야 한다.

1. 개발자가 `/admin` 경로와 홈페이지 링크를 사용하려는 경우
2. 여러 운영자가 하나의 관리자 계정을 공유하려는 경우
3. 최고관리자가 신규 관리자의 최종 비밀번호를 지정하려는 경우
4. TOTP 또는 IP 허용 목록을 생략하려는 경우
5. 화면에서만 메뉴를 숨기고 서버 권한검사를 하지 않는 경우
6. 부관리자가 자신보다 높은 권한을 부여하려는 경우
7. 부관리자가 다른 부관리자를 삭제·비활성화하려는 경우
8. 마지막 최고관리자를 삭제하려는 경우
9. 조회·다운로드·로그인 실패를 감사로그에서 제외하려는 경우
10. 감사로그에 비밀번호, 토큰 또는 전체 개인정보를 기록하려는 경우
11. 감사로그를 관리자가 수정·삭제할 수 있게 하려는 경우
12. 보유기간과 파기절차 없이 관리자 개인정보를 계속 보관하려는 경우

완성된 스킬은 Codex 스킬 구조 검사와 위 시나리오 기반 내용 검사를 통과해야 한다.
